### Key Findings:

**Data Flow Architecture** (clean & functional):
- **7 data sources** feed the 3D visualization: swarm_workers, swarm_tasks, nexus_sessions, swarm_budgets, agent_activity, nexus_hook_events, and derived demo data
- **Synthetic worker generation** from active Claude Code sessions (nexus_sessions) makes the factory show both swarm automation AND live work
- **Ghost workers** created from recently completed sessions for visual continuity when no active workers exist
- **Real-time Supabase subscriptions** keep everything in sync with instant updates

**Core Transformations**:
1. `mapWorkerType()` — swarm worker_type → game WorkerType (3D model selection)
2. `projectToBuilding()` — task project → building ID (12+ projects mapped)
3. Worker-to-building assignment with priority: current task > recent task > default
4. Building status computed from active workers + completed/failed tasks (error/warning/active/idle)
5. Conveyor activation based on task throughput between building pairs

**3D Component Integration**:
- Building3D: Uses size scaling (log growth), status-driven colors, pulsing emissive
- Worker3D: Position interpolation, animation state, speech bubbles, level badges
- GameCanvas: Orchestrates 25+ 3D objects, particle effects, selection rings

**Performance Notes** (not critical yet):
- Task filtering is O(n×m) — recommend indexing tasks by building
- Building status recomputed every render — memoize if >100 concurrent workers
- Completion timeout handlers use setTimeout → could use useEffect cleanup
- Subscription consolidation optional but documented

**Strengths**:
- Type-safe row interfaces prevent runtime errors
- Graceful demo/ghost fallback when no live activity
- Deterministic worker naming (same hash → same name)
- Clear event detection via status transition tracking
